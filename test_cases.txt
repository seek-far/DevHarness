  实际数字

  evaluation 总共 ~955 LOC，5 个测试 case ——case 密度最低的一块。bf_worker 那边 ~3000 LOC 对应 200+ case。

  风险排序（哪些缺失最该补）

  🔴 metrics.compute_metrics —— 5 行就能写一个，但聚合逻辑里"outcome=fixed 的占比"、"avg over None elapsed_s"、"空
  records 列表" 都是会出错的边界，目前如果指标数表错了，没人会发现。

  🔴 runner.run_sweep —— 决定 RunRecord 形状的核心。我们这次几次改动都加了 RunRecord 字段（parse_trace_fallback /
  source_fetch_failed / 5 个 retry counter），如果 runner 漏掉某个字段就只能等下一次跑完才发现。一个最小测试：用 stub
  Agent + 1 个 fixture 跑一轮，验证生成的 RunRecord 字段集 = RunRecord dataclass 字段集。

  🟡 fixture.Fixture loader —— 加载是简单的 JSON 读，但有"meta.json 缺字段"、"trace.txt 不存在"、"category 字段缺失"这些
   schema 边界。

  🟡 promote 剩下的分支 —— 当前的 2 case 只覆盖了 happy path。base_commit_from_record 在 branch_create_result /
  commit_result 多重回退、URL 不存在、commit 在远端没找到等情况下的行为没测。

  🟢 cli.py —— 是命令行壳子，主要工作 delegate 给上面这些模块。本身覆盖价值不高，靠 bash 烟测就够。

  🟢 coordinator.py 的剩余部分（subprocess + worktree） —— 测起来麻烦（要 mock subprocess.run + git
  worktree），收益相对有限。

  建议的最小补丁

  如果只补一组，我会建议这两个：

  1. tests/test_metrics.py —— 直接构造若干 RunRecord dict 喂进 compute_metrics，验证
  fix_rate、avg_iterations、avg_elapsed_s。也包括空 list、全 error、混合 outcome。约 6-8 个 case，纯单元，不需要 mock。
  2. tests/test_runner_record_completeness.py —— 一个"防漏字段"的 smoke 测试：stub Agent + 临时 fixture 目录，跑
  run_sweep 一格，断言生成的 RunRecord JSON 里包含 RunRecord.__dataclass_fields__ 的所有 key。这样以后再加 RunRecord
  字段没接到 runner 时会立刻报红。1-2 个 case，归到 category 3（用 stub Agent）。
